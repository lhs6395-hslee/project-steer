**적용 범위**: 전체 레이아웃 (L01~L36). 재설계 아님 — 기존 산출물 품질 검증 기준.
**갱신**: 2026-04-16

## 0. 한국 IT 업계 맥락

이 문서는 **한국 IT 기업의 실무 보고 자료**를 대상으로 한다.
서구 프레젠테이션 이론(TED 스타일, Duarte 방법론)은 참고 원칙이지만, 한국 업계 관행이 우선한다.

### 한국 IT 보고서의 구조적 특징

| 특징 | 설명 |
|------|------|
| **문서 완결성** | 발표자 없이 문서만으로 읽혀야 함. 구두 보완 의존 금지 |
| **고밀도 정보** | 빈 여백은 미완성으로 인식. 카드·도형·텍스트로 슬라이드를 채움 |
| **수치 기반 신뢰** | 정성적 서술보다 정량 수치, 출처, 날짜가 포함될수록 신뢰도 상승 |
| **계층적 독자** | 임원(결론·숫자)과 실무자(근거·상세) 모두를 한 문서에서 만족 |
| **RFP 경쟁 환경** | 평가위원이 수십 개 제안서를 검토 — 첫 인상과 구조의 명확성이 차별 요인 |
| **보수적 색상 선호** | 감각적 색상보다 파랑·회색 기반의 안정적 팔레트 선호 |
| **아이콘·다이어그램 필수** | 텍스트만의 슬라이드는 낮은 품질로 평가됨 |

### 이 문서의 원칙 적용 방식

- **서구 이론 원칙**: 대비비율, 시선 흐름, 색상 접근성은 그대로 적용
- **한국 관행 우선 원칙**: 정보 밀도, 여백 처리, 문서 완결성은 한국 관행 기준
- 두 원칙이 충돌할 경우 → **한국 관행 우선, 단 가독성(대비비율)은 양보 불가**

## 2. 슬라이드 유형별 목적과 원칙

한국 IT 업계의 주요 슬라이드 유형은 세 가지로 구분된다:

| 유형 | 대상 독자 | 핵심 목적 | 정보 밀도 |
|------|---------|---------|---------|
| **임원 보고** | C-레벨, 본부장 | 의사결정 지원 | 낮음 — 핵심만 |
| **RFP 제안서** | 평가위원회 | 경쟁 우위 입증 | 높음 — 근거 밀도 |
| **결과 보고서** | 실무 팀장·담당자 | 진행 현황·성과 공유 | 중간 — 데이터 중심 |
| **주간 보고서** | 팀장·PM·고객사 담당자 | 진척 현황·이슈·다음 계획 | 중간 — 표+수치 중심 |
| **아키텍처 보고서** | 기술 리뷰어·아키텍트·고객 CTO | 기술 구조 설명·검토 | 높음 — 다이어그램 필수 |
| **서비스 가이드** | 운영자·사용자·파트너 | 기능 설명·운영 절차 안내 | 중간 — 스텝바이스텝 |

### 유형별 구성 원칙

#### 주간 보고서

**목적**: "이번 주 무슨 일이 있었고, 무엇이 문제이며, 다음 주 무엇을 할 것인가"를 1장~3장으로 전달.

**필수 구성 요소:**
- **진척 현황**: WBS 기준 완료율 (숫자/% 필수, "진행 중" 단독 표현 금지)
- **완료 사항**: 이번 주 완료된 항목 (측정 가능한 결과로 표현)
- **이슈/리스크**: 발생 이슈, 영향도, 조치 계획 (색상 Semantic — SEM_RED/YELLOW)
- **다음 주 계획**: 구체적 액션 + 담당자 + 목표일

**레이아웃 권장**: L05 Phased Columns(단계별), L03 Grid 2×2(항목별), L04 Process Arrow(프로세스)
**금지**: "순조롭게 진행 중" 등 정량화 없는 서술. 이슈 없음 슬라이드에도 "리스크 없음 확인" 명시.

#### 아키텍처 보고서

**목적**: 기술 구조를 비기술 임원과 기술 검토자 모두가 이해할 수 있도록 레이어별로 설명.

**필수 구성 요소:**
- **전체 아키텍처 다이어그램**: 1장 — 컴포넌트·연결·데이터 흐름 표시
- **레이어별 상세**: 각 레이어(프론트엔드/백엔드/인프라/데이터)별 1~2장
- **기술 스택 근거**: 왜 이 기술을 선택했는가 (대안과 비교 포함)
- **비기능 요건**: 가용성·확장성·보안·비용 수치 명시

**레이아웃 권장**: L01 Bento Grid(다이어그램+설명), L25 Architecture Wide(전체 구조도), L09 Challenge/Solution(As-Is→To-Be)
**다이어그램 원칙**: 컴포넌트는 도형으로, 연결은 화살표로, 외부 시스템은 별도 색상(SUB_BLUE). 텍스트만으로 아키텍처 설명 금지.

#### 서비스 가이드

**목적**: 독자가 문서만으로 서비스를 설치·설정·운영할 수 있도록 단계별 안내.

**필수 구성 요소:**
- **전제 조건**: 사전 준비 사항 명시 (버전, 권한, 환경)
- **단계별 절차**: 번호 있는 순서 (Step N 형식)
- **스크린샷/다이어그램**: 각 주요 단계마다 시각 자료 포함
- **확인 방법**: 각 단계 완료 후 "정상 확인" 방법 명시
- **트러블슈팅**: 주요 오류 코드 + 해결 방법

**레이아웃 권장**: L04 Process Arrow(절차), L07 Timeline(단계별 흐름), L16 Detail Image(스크린샷+설명)
**금지**: 단계 없이 텍스트만 나열. 스크린샷 없는 UI 설명. "적절히 설정한다" 등 모호한 지침.

## 3. 핵심 디자인 원칙

### 3.1 정보 구조 — Minto Pyramid Principle

**[외부]** Barbara Minto, *The Pyramid Principle*, McKinsey & Company

> "결론을 먼저 말하고, 그 근거를 MECE하게 제시하라."

- **SCQ 구조**: Situation(현황) → Complication(문제) → Question(질문) → Answer(결론)
- **슬라이드 제목 = 결론**: "클라우드 비용 35% 절감 가능" (사실 서술 X, 인사이트 O)
- **MECE 원칙**: 카드/항목이 서로 겹치지 않고 전체를 포괄
- **적용**: 임원 보고 슬라이드의 중제목은 반드시 결론형 문장으로 작성

### 3.2 시선 흐름 — F패턴 / 레이어케이크 패턴

**[외부]** Nielsen Norman Group, *Text Scanning Patterns: Eyetracking Evidence* (2023)

> "독자는 모든 단어를 읽지 않는다. 상단·좌측을 우선 스캔하고, 헤딩을 레이어케이크처럼 훑는다."

**단, 한국 IT 업계 보고서는 이 원칙을 그대로 적용하지 않는다:**

| 서구 프레젠테이션 이론 | 한국 IT 업계 실제 관행 |
|-------------------|-------------------|
| 1슬라이드 1메시지 | 관련 정보를 한 슬라이드에 묶어서 전달 |
| 여백(White Space) 강조 | 빈 여백은 "미완성"으로 인식 — 채워야 함 |
| 텍스트 최소화 | 근거·수치·출처 텍스트가 많을수록 신뢰도 상승 |
| 발표자가 구두로 보충 | 문서 자체로 완결 — 발표 없이도 읽혀야 함 |
| 심플하고 여유 있는 레이아웃 | 카드·도형·아이콘으로 공간을 최대한 활용 |

**따라서 이 프로젝트의 적용 원칙:**
- **빈 공간 금지**: 카드·패널이 있으면 반드시 내용을 채운다
- **헤딩 위계는 유지**: 글자가 많더라도 제목(14pt Bold) → 본문(13pt) 위계로 스캔 가능하게
- **좌상단 최우선 원칙은 유지**: 가장 중요한 메시지는 항상 좌상단 또는 첫 번째 카드
- **5-second test 변형 적용**: 5초 안에 "이 슬라이드가 무엇에 관한 것인지"는 알 수 있어야 함 (세부 내용은 이후 읽기)

### 3.3 시각 위계 — 3단계 크기 규칙

**[외부]** Interaction Design Foundation, *Visual Hierarchy* (2025); Nielsen Norman Group, *Visual Hierarchy in UX* (2024)

> "크기는 중요도를 나타낸다. 위계는 3단계를 초과하지 않는다."

| 위계 | 용도 | 폰트 | 크기 | 색상 |
|------|------|------|------|------|
| 0단계 | 중제목 (슬라이드 좌상단 헤더) | Freesentation Bold | 20pt | BLACK `#000000` |
| 0단계 보조 | 중제목 설명글 (헤더 우측) | Freesentation | 12pt | GRAY (테마 상속) |
| **1단계** | **본문제목** (구분선 아래, 콘텐츠 상단) | **Freesentation Bold** | **16pt** | **PRIMARY `#0043DA`** |
| **1단계 보조** | **본문제목 설명글** (본문제목 바로 아래) | **Freesentation Regular** | **13pt** | **DARK_GRAY `#212121`** |
| 2단계 | 카드/도형 제목 | 프리젠테이션 7 Bold | 14pt | PRIMARY 또는 DARK_GRAY |
| 3단계 | 카드/도형 본문·설명 | Freesentation Regular | 13pt | DARK_GRAY `#212121` |

- **Bold 최대 2요소/슬라이드**: 강조가 너무 많으면 강조가 없는 것과 같음
- **색상 위계**: PRIMARY(파랑) = 핵심 강조, DARK_GRAY = 본문, GRAY = 보조
- **Squint Test**: 눈을 가늘게 떴을 때도 중요 요소가 먼저 보여야 함

### 3.4 인지 부하 감소 — 시각화 우선

**[외부]** Nielsen Norman Group, *Minimize Cognitive Load* (2024); Stephen Few, *Show Me the Numbers* (2012)

> "텍스트로 설명할 수 있는 것을 다이어그램으로 보여줄 수 있다면, 다이어그램이 우선이다."

- **텍스트 → 시각화 치환 우선순위**:
  1. 프로세스/흐름 → 화살표 다이어그램 (L04 Process Arrow)
  2. 비교 → 카드 그리드 (L02 Three Cards, L03 Grid 2×2)
  3. 단계/로드맵 → 컬럼 레이아웃 (L05 Phased Columns)
  4. 시계열 → 타임라인 (L06 Zigzag Timeline, L07 Timeline)
- **불필요한 잉크 제거**: 장식용 선, 그림자, 3D 효과 금지 (Few의 Data-Ink Ratio)
- **불릿 포인트 제한**: 텍스트가 카드/도형 경계를 넘치지 않는 한 개수 제한 없음. 넘칠 경우 카드를 분리하거나 폰트 축소 (최소 13pt)

### 3.5 정보 밀도 — 한국 IT 업계 기준

**[추측 + 실무 관행]** 한국 IT 기업의 보고·제안 문서는 발표 없이 단독으로 유통되는 경우가 많음

한국 IT 보고서는 서구 프레젠테이션 이론과 다르게 작동한다:

| 서구 이론 | 한국 IT 실무 |
|---------|-----------|
| 1슬라이드 1메시지 | 관련 메시지 묶음 — 슬라이드 수 절약 |
| 텍스트 최소화 | 근거·수치·출처 텍스트가 많을수록 신뢰도 ↑ |
| 여백(White Space) 권장 | 빈 여백 = 미완성 인식. 채워야 함 |
| 발표자가 구두로 보충 | 문서 자체 완결 — 배포 후 단독 열람 |
| 심플·미니멀 디자인 선호 | 카드·도형·아이콘·다이어그램으로 공간 활용 |
| 1컬럼 레이아웃 | 2~4컬럼 분할 레이아웃 선호 |

**적용 규칙:**
- **빈 카드·패널 금지**: 레이아웃이 N개 카드면 N개 모두 내용 채움
- **수치 필수**: "비용 절감" X → "연간 ₩3.2억 절감 (ROI 240%)" O
- **출처·기준일 표기**: 모든 수치 옆에 출처 또는 기준일 명시
- **아이콘 필수**: 카드·패널이 있으면 아이콘 반드시 포함. 파란 원형 대체 금지. **예외**: 같은 슬라이드 내에 흐름도·아키텍처 다이어그램이 시각 요소를 충분히 제공하는 경우 카드 아이콘 생략 가능
- **요약 텍스트 필수**: 슬라이드 콘텐츠 영역 상단에 핵심 메시지 1~2줄 (16pt PRIMARY)
- **단, 가독성 양보 불가**: 정보 밀도를 높이더라도 대비비율 4.5:1, 폰트 13pt 이상 유지

### 3.6 RFP 제안서 특화 원칙

**[외부]** 조달청 *국가계약법 기반 RFP 평가 기준* (2024); 행정안전부 *공공 SW사업 제안요청서 가이드* (2024)

평가위원이 중점 심사하는 항목:

1. **문제 정의 명확성**: 고객 현황과 문제를 정확히 이해했는가
2. **해결 방안 구체성**: 추상적 방향이 아닌 구체적 실행 계획
3. **실현 가능성**: 일정·인력·비용 근거가 충분한가
4. **차별성**: 경쟁사 대비 우리만의 강점이 명확한가
5. **레퍼런스**: 유사 사업 수행 실적

**슬라이드 구성 원칙:**
- 각 슬라이드가 위 5개 항목 중 하나를 명확히 답해야 함
- 제안서 표지: 고객명·사업명·제안사명·날짜 4요소 필수
- 페이지 번호 + 섹션명을 상단 바에 항상 표시 (현재 템플릿의 placeholder 활용)

### 3.7 임원 보고 특화 원칙

**[외부]** Gartner, *"How to Present to Senior Executives"* (2024); Barbara Minto, *Pyramid Principle*

> "임원은 3분 안에 의사결정을 내려야 한다. 슬라이드는 그 판단을 지원하는 도구다."

- **BLUF (Bottom Line Up Front)**: 첫 슬라이드에 결론·권고안을 먼저 제시
- **3분 룰**: 임원 보고는 핵심 슬라이드 3장 이내로 요약 가능해야 함
- **Action Item 명시**: 보고 후 무엇을 결정/승인해야 하는지를 슬라이드에 명시
- **데이터 출처 표기**: 모든 수치에 출처·기준일 표기 (신뢰성 확보)

## 4. 색상 선택 기준 (공식 문서 기반)

### 4.1 대비비율 — 절대 기준

**[외부]** W3C, *WCAG 2.1 SC 1.4.3 Contrast (Minimum)* (2023)
**[외부]** Microsoft, *"Make your PowerPoint presentations accessible"* (2025)
**[외부]** Microsoft, *"Design Power BI Reports for Accessibility"* (2026-01)

> "텍스트와 배경의 대비비율이 4.5:1 미만이면 저시력·고령 독자가 읽을 수 없다."

| 기준 | 대비비율 | 적용 대상 |
|------|---------|---------|
| WCAG AA (필수) | **4.5:1 이상** | 일반 텍스트 (13~14pt) |
| WCAG AA 대형 텍스트 | **3:1 이상** | 18pt 이상 또는 14pt Bold |
| WCAG AAA (권장) | **7:1 이상** | 임원·RFP 문서 최고 품질 목표 |

**왜 하늘색 배경 + 흰색 텍스트가 안 되는가:**

```
하늘색 (예: #87CEEB, RGB 135,206,235) vs 흰색 (#FFFFFF)
→ 대비비율 약 1.6:1 ← WCAG AA 기준(4.5:1)의 1/3 수준
→ 프로젝터·밝은 조명 환경에서는 사실상 불가독
→ 색맹(청색 색약) 독자에게는 배경과 텍스트 구분 불가
```

**실무 합격선 조합 (실측 대비비율):**

| 배경 | 텍스트 | 대비비율 | 판정 |
|------|-------|---------|------|
| 흰색 `#FFFFFF` | 검정 `#000000` | 21:1 | ✅ AAA |
| 흰색 `#FFFFFF` | PRIMARY `#0043DA` | 7.2:1 | ✅ AAA |
| 흰색 `#FFFFFF` | DARK_GRAY `#212121` | 16.1:1 | ✅ AAA |
| BG_BOX `#F8F9FA` | PRIMARY `#0043DA` | 6.9:1 | ✅ AA |
| PRIMARY `#0043DA` | 흰색 `#FFFFFF` | 7.2:1 | ✅ AAA (배지용) |
| 하늘색 `#87CEEB` | 흰색 `#FFFFFF` | 1.6:1 | ❌ 실격 |
| 연회색 `#DCDCDC` | 흰색 `#FFFFFF` | 1.3:1 | ❌ 실격 |
| 노랑 `#FFFF00` | 흰색 `#FFFFFF` | 1.07:1 | ❌ 실격 |

### 4.2 금지 색상 조합

**[외부]** Microsoft, *"Design Power BI Reports for Accessibility"* (2026-01)

Microsoft 공식 문서가 명시한 색맹·저시력 독자가 구분하기 어려운 조합:

| 금지 조합 | 이유 |
|---------|------|
| 초록 + 빨강 | 적록색맹(남성 8%) 구분 불가 |
| 초록 + 갈색 | 유사 색조로 혼동 |
| 파랑 + 보라 | 명도 차이 부족 |
| 초록 + 파랑 | 청록 혼동 |
| 연두 + 노랑 | 명도 차이 부족 |
| 파랑 + 회색 | 채도 차이 부족 |
| 초록 + 회색 | 채도 차이 부족 |
| 초록 + 검정 | 명암 역전 시 구분 불가 |

**적용 규칙**: 위 조합은 카드 배경색, 배지 색상, 강조 텍스트에 사용 금지.

### 4.3 색상 선택 4단계 프로세스

**[외부]** Microsoft Power BI Accessibility Guide (2026); WCAG 2.1

1. **배경색 먼저 확정** → 슬라이드 배경(흰색/BG_BOX)을 기준점으로 설정
2. **텍스트 대비 검증** → 배경 대비 4.5:1 이상인 텍스트 색만 사용
3. **색맹 시뮬레이션** → 초록·빨강 조합 회피, 명도(Lightness) 차이로 구분
4. **Squint Test** → 눈을 가늘게 떴을 때 텍스트가 배경과 분리되어 보이는지 확인

**도구**:
- WebAIM Contrast Checker: `webaim.org/resources/contrastchecker`
- Microsoft Accessibility Checker (PowerPoint 내장 — 검토 탭)

### 4.4 이 프로젝트 팔레트의 대비비율 검증

`pptx_style_guide.md`의 COLORS dict 기준 사전 검증 결과:

| Key | RGB | 흰 배경 대비 | BG_BOX 대비 | 판정 |
|-----|-----|-----------|-----------|------|
| PRIMARY | (0,67,218) | 7.2:1 | 6.9:1 | ✅ AAA / AA |
| DARK_GRAY | (33,33,33) | 16.1:1 | 15.5:1 | ✅ AAA |
| GRAY | (80,80,80) | 7.0:1 | 6.7:1 | ✅ AA |
| SUB_BLUE | (42,111,222) | 5.1:1 | 4.9:1 | ✅ AA |
| SUB_ORANGE | (238,129,80) | 2.7:1 | 2.6:1 | ❌ 소형 텍스트 불가 (아이콘·배지 대형만 허용) |
| SUB_GREEN | (76,184,143) | 2.5:1 | 2.4:1 | ❌ 소형 텍스트 불가 |
| BORDER | (220,220,220) | 1.6:1 | — | 텍스트 색으로 사용 금지 |

### 4.5 배경색 선택 기준

**[외부]** Microsoft, *"Make your PowerPoint presentations accessible"* (2025)

> "White and black schemes make it easier for people who are colorblind to distinguish text and shapes."

| 상황 | 권장 배경 | 이유 |
|------|---------|------|
| 본문 슬라이드 | 흰색 `#FFFFFF` | 최고 대비, 인쇄·프로젝터 모두 안전 |
| 카드/패널 배경 | BG_BOX `#F8F9FA` | 흰 배경과 구분, 텍스트 대비 유지 |
| 강조 배지 | PRIMARY `#0043DA` + 흰 텍스트 | 7.2:1 — AAA 통과 |
| **금지** | 채도 높은 유색 배경 + 흰 텍스트 | 대비 붕괴 위험 (하늘색·연두·노랑 등) |
| **금지** | 어두운 배경 + 어두운 텍스트 | 명도 차이 부족 |

### 4.6 색상 수 제한

**[외부]** Nielsen Norman Group, *Visual Hierarchy in UX* (2024); Microsoft Power BI Guide (2026)

> "Limit to 2 primary + 2 secondary colors for simple designs. Use no more than 3 contrast variations."

- **슬라이드 1장당 강조색 최대 2종** (PRIMARY + 서브 1종)
- **그라데이션**: 단계 표현(L05처럼 의미가 있는 경우)에만 허용. 장식용 그라데이션 금지
- **색만으로 의미 전달 금지**: 색 외에 아이콘·텍스트·형태로 보완 (색맹 대응)

## 7. 금지 사항 (Anti-patterns)

**[외부]** Stephen Few, *Show Me the Numbers* (2012); NNG, *Minimize Cognitive Load* (2024)

| 금지 | 이유 |
|------|------|
| 3D 차트·그림자·글로우 효과 | 데이터 왜곡, 인지 부하 증가 |
| 슬라이드당 폰트 3종 이상 | 시각적 혼란 |
| 이미지 없는 전체 텍스트 슬라이드 | 스캔 불가, 임원 이탈 |
| 출처 없는 수치 | 신뢰성 손상 (RFP 감점 요인) |
| 파란 원형 아이콘 대체 | 비전문적 완성도 |
| 아이콘을 패널/도형 경계에 타이트하게 붙이기 | 시각적 압박감 — 패널 내부 여백 최소 0.15" 확보 필수 |
| 아이콘이 패널 경계 밖으로 넘침 | 레이아웃 붕괴 — `아이콘_right ≤ 패널_right - 0.15"` 항상 검증 |
| 텍스트 TextBox bottom이 아이콘 top을 초과 | 텍스트-아이콘 겹침 — 동적으로 TextBox height 조정 필요 |
| 카드 내 빈 내용 | 정보 밀도 부족 |
| 제목만 있고 내용 없는 슬라이드 | "추후 작성" 슬라이드 금지 |
| 카드/도형 경계를 넘치는 텍스트 | 카드 분리 또는 폰트 축소 (최소 13pt) — 개수 자체는 제한 없음 |
