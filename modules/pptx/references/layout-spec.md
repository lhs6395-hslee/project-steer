# PPTX 레이아웃 스펙

템플릿: `modules/pptx/templates/pptx_template.pptx`
슬라이드 크기: 12192000 × 6858000 EMU (13.333" × 7.500")
총 슬라이드: 41장 (템플릿 기본)

## 슬라이드 구조 규칙

```
표지 (1페이지) → 목차 (2페이지~, 5개씩 페이징) → 본문 N장 → 끝맺음 (41페이지)
```

- 표지: 항상 1페이지 (슬라이드 0)
- 목차: 2페이지부터 시작. 한 페이지에 최대 5개 섹션. 6개 이상이면 새 목차 페이지 추가.
- 본문: 목차 다음부터 순차 배치
- 끝맺음: 항상 41페이지 (템플릿 마지막 슬라이드, index 40) 사용

### 표지 텍스트 교체 규칙

- **서식 보존 필수**: `tf.clear()` + 새 run 생성 금지. 기존 run의 XML을 복제하여 텍스트만 교체 (`replace_text_preserve_format` / `replace_multiline_preserve_format` 사용)
- 이유: 템플릿의 scheme color (BACKGROUND_1 = 흰색), 폰트명, 크기가 보존되어야 함
- `tf.clear()`를 사용하면 scheme color가 사라지고 검정색으로 렌더링됨
- shape[3], [4] (회사 소개)는 수정하지 않음 (원본 유지)

### 프레젠테이션 섹션(Section) 관리

### 목차 텍스트 교체 규칙

- **서식 보존 필수**: 표지와 동일하게 `replace_multiline_preserve_format` 사용. scheme color, 폰트명, 크기 보존.
- **5줄 패딩**: 템플릿 원본이 5줄 구조이므로, 섹션이 5개 미만이면 빈 문자열("")로 나머지 줄 채움
- **섹션명 자동 너비**: `auto_fit_textbox_width`로 가장 긴 섹션명에 맞춰 텍스트박스 너비 자동 조정 (24pt 기준)
- **번호 열**: 서식 보존, alignment CENTER 유지

### 목차 페이징 구현

6개 이상 섹션 시 추가 목차 페이지를 생성한다.

**방식**: 첫 번째 목차 슬라이드의 spTree 자식 요소를 통째로 복제하여 새 슬라이드에 삽입. 이렇게 하면 CONTENTS 헤더, 장식 그룹, 번호/섹션명 서식이 모두 보존된다.

```
1. add_slide(layout_index=4)로 빈 목차 슬라이드 생성
2. 새 슬라이드의 spTree에서 기존 자식 제거 (grpSpPr 제외)
3. 첫 번째 목차의 spTree 자식을 deepcopy하여 새 spTree에 추가
4. shape[1](번호열), shape[2](섹션명열) 텍스트만 replace_multiline_preserve_format으로 교체
5. shape[2]에 auto_fit_textbox_width 적용
```

**주의**: `add_slide` 후 shape를 직접 추가하는 방식(TextBox 생성)은 CONTENTS 헤더, 장식 그룹 등이 누락되므로 사용하지 않는다.

페이징 규칙:
- 섹션 1~5: 첫 번째 목차 페이지
- 섹션 6~10: 두 번째 목차 페이지 (동일 레이아웃으로 새 슬라이드 추가)
- 섹션 11~15: 세 번째 목차 페이지 ...
- 각 페이지의 번호 열은 해당 페이지의 시작 번호부터 (예: 6\n7\n8\n9\n10)

| 요소 | left | top | width | height | 용도 |
|------|------|-----|-------|--------|------|
| placeholder 상단바 | 354806 | 101016 | 2807494 | 250530 | 상단 바 (수정 불필요) |
| **제목 TextBox** | 354806 | 557906 | 2720317 | 932614 | "N-N. 제목" (좌측, 멀티라인 허용 최대 3줄) **margin=0 필수** |
| **설명 TextBox** | 3763700 | 558800 | 7551483 | 937918 | 설명글 (우측) **margin=0 필수** |

**중제목 TextBox margin 설정 (CRITICAL):**
```python
text_frame.margin_left = 0
text_frame.margin_right = 0
text_frame.margin_top = 0
text_frame.margin_bottom = 0
```
python-pptx는 TextBox 생성 시 자동으로 margin 설정 (L/R: 91440 EMU = 0.1", T/B: 45720 EMU = 0.05"). 이를 명시적으로 0으로 설정해야 텍스트가 정확한 위치에 표시됨. margin이 있으면 TextBox 내부에서 텍스트가 밀려나 레이아웃 불일치 발생.

본문 제목은 28pt에서 자동 줄바꿈으로 2~3줄이 될 수 있다. 텍스트박스 높이(932614 EMU = 73.4pt)는 28pt × 1.2 × 2.18줄 수용 가능. 3줄이 필요한 경우 높이를 1050000 EMU(≈82.7pt)로 확장한다.

본문 콘텐츠 영역: top 1828800 ~ 6580000 (약 2.0" ~ 7.2")

## 디자인 원칙 (합의됨 — 모든 레이아웃 적용)

### 품질 기준

- **정보 과밀 선호**: 빈 공간보다 밀도 높은 레이아웃을 지향. 카드/패널이 있으면 내용을 채운다
- **전문가 수준 가시성**: 컨설팅 자료 수준의 완성도. 텍스트 정렬, 여백, 색상이 일관되어야 함
- **복합 레이아웃 우선**: 단순 텍스트보다 다이어그램+카드, 플로우+설명 등 복합 구성 선호
- **아이콘 필수**: 카드/패널이 있으면 아이콘을 반드시 포함. 파란 원형 대체 금지 (실제 PNG 사용)
- **아이콘 사용 전 시각 확인 필수**: `icons/png/` 파일은 파일명과 내용이 일치하지 않을 수 있음. Read tool로 이미지를 열어 내용 확인 후 사용.
- **슬라이드 간 이미지 공유 주의**: `ppt/media/` 파일 삭제/교체 전 전체 rels 참조 확인. 다른 슬라이드가 같은 파일을 참조할 수 있음.

### 영역 구분 (겹침 절대 금지)

```
┌─────────────────────────────────────────────┐
│ 상단 바 (y=0~0.44")  — slideLayout 상속      │
│ 중제목 영역 (y=0.61~1.63") — 절대 침범 금지  │ ← 중제목 bottom
├─────────────────────────────────────────────┤ ← 본문 시작 최소 y=1.75"
│ 본문 콘텐츠 영역 (y=1.75"~7.00")            │
└─────────────────────────────────────────────┘
```

**본문 shape top 최솟값: 1.75"** — 이 값보다 위에 shape 배치 시 중제목과 겹침 발생.

### 본문 콘텐츠 배치 규칙

- **중앙 정렬 필수**: 다이어그램, 카드 그룹, 리스트 등 본문 콘텐츠 블록은 슬라이드 좌우 중앙에 배치
  - 계산: `start_left = (SLIDE_W - total_block_width) / 2`
- **도형 제목**: 프리젠테이션 7 Bold, 14pt, PRIMARY 색상
- **도형 내용**: Freesentation, 13pt, DARK_GRAY 색상
- **좌우 여백 대칭**: 왼쪽 여백 = 오른쪽 여백 (균등)
- **상하 여백 균등 (필수)**: 실질 본문 구역 기준으로 콘텐츠 상단 여백 = 하단 여백. 아래 분기로 계산:

  ```
  분기 A — 본문 제목 + 본문 설명글이 있는 레이아웃 (L01~L05):
    실질 본문 구역 top = 본문 설명글 bottom (통상 2.45")
    실질 본문 구역 bot = 슬라이드 하단 (7.50")
    목표 여백 = (콘텐츠_top - 2.45" + 7.50" - 콘텐츠_bot) / 2
    shift_y   = 목표 여백 - (콘텐츠_top - 2.45")

  분기 B — 본문 제목/설명글 없이 중제목만 있는 레이아웃 (L06 등):
    실질 본문 구역 top = slideLayout 구분선 y (1.6368")
    실질 본문 구역 bot = 슬라이드 하단 (7.50")
    목표 여백 = (콘텐츠 전체 높이를 구역 안에서 상하 균등 배치)
    목표 top  = 1.6368" + (7.50" - 1.6368" - 콘텐츠_높이) / 2
  ```

  구분선 y 실측값: slideLayout2 `직선 연결선 5` top = **1.6368"** (1,497,492 EMU)

여백 실측값: § "본문 레이아웃 공통 원칙" 여백 테이블 참조

- **도형 내부 텍스트**: 배지/단계명 등 도형 단일 텍스트는 shape의 text_frame에 직접 삽입. bodyPr inset(tIns/bIns/lIns/rIns)으로 상하좌우 내부 여백 균등 설정 (0.1" 권장). 별도 TextBox 올리기 금지
- **텍스트 수직 정렬**: 도형 내 단일 텍스트는 `anchor='ctr'`(Middle)로 수직 중앙. 제목+본문 복합 텍스트는 상단 정렬 + inset 패딩
- **요약 텍스트 영역**: 콘텐츠 top 1.75" / 2.15"
  - **본문제목**: `Freesentation` 16pt PRIMARY Bold
  - **본문설명**: `Freesentation` 13pt DARK_GRAY Regular
  - ⚠ **중제목(Freesentation 20pt 검은색)과 크기/폰트 다름** — 본문제목/설명은 PRIMARY/DARK_GRAY 색상
- **요소 간 최소 간격**: 0.2"
- **요소 겹침 금지**: 텍스트가 텍스트박스를 넘어서 다른 요소와 겹치지 않도록 사전 검증
- **카드 내부 요소 범위 검증**: CardTitle, CardBody, Icon은 반드시 부모 Card의 y ~ y+h 범위 안에 있어야 함. 카드 y를 변경할 때 내부 요소도 반드시 같이 이동. 검증식: `card_top ≤ element_top AND element_bottom ≤ card_bottom`

## 본문 레이아웃 상세: Zigzag Timeline (`zigzag_timeline`) — L06

지그재그 N노드 타임라인. 중앙 수평선 위아래로 카드가 교대 배치.
홀수 노드(1,3,5...) = 카드 위쪽, 짝수 노드(2,4,6...) = 카드 아래쪽.
한국 IT 스타일 — 빈 여백 없음, 각 노드마다 3 bullet, 아이콘 필수.
**노드 수**: 3~6개 (기본 4개). steps 배열 길이로 자동 결정.

참조: design-spec.md § 3.2 (시선흐름 — S자 지그재그 패턴), § 3.4 (시각화 우선 — 선형 프로세스를 시각적 흐름으로), § 3.5 (고밀도 — 4노드×3 bullet=12항목 단일 슬라이드), § 3.8 (NIA 계획단계 — 프로젝트 타임라인 명시)

**분기 A** — 본문 제목 + 본문 설명글이 있는 레이아웃.
실질 본문 구역 top = 본문 설명글 bottom (2.45"). zigzag 모든 요소는 y ≥ 2.450" 배치.
좌/우 여백: left=0.50", right=0.50" (균등). 상/하 여백 목표: 0.425" (콘텐츠를 구역 안에서 균등 배치).

#### N노드 동적 좌표 계산 (N ≠ 4)

위 좌표 테이블은 4노드 기준. steps 배열이 3~6개일 때 동적 계산:

```python
N = len(steps)                    # 3~6
MARGIN = 0.50                     # 좌/우 여백 (인치)
SLIDE_W = 13.333
NODE_W = 0.60
AVAILABLE = SLIDE_W - 2 * MARGIN  # 12.333"
gap = (AVAILABLE - NODE_W) / (N - 1)

for i in range(N):
    node_left = MARGIN + i * gap + (gap - NODE_W) / 2
    card_w = min(2.40, gap - 0.50)  # 최소 간격 0.50" 확보
    card_left = node_left + NODE_W / 2 - card_w / 2
    position = "above" if i % 2 == 0 else "below"
```

| N | 노드 간격 | 카드 너비 | 비고 |
|---|----------|----------|------|
| 3 | 5.87" | 2.40" (최대) | 여유 공간 넓음 |
| 4 | 3.91" | 2.40" (최대) | 기본값 — 위 좌표 테이블 참조 |
| 5 | 2.93" | 2.40" | 간격 충분 |
| 6 | 2.27" | 1.77" | 카드 축소 필요 |

### 배지-패널 분리 원칙 (핵심)

```

배지 top  = 본문설명글 bottom(2.45") + 상단 여백(0.2") = 2.65"
배지 bottom = 2.65" + 0.3" = 2.95"
패널 top  = 배지 bottom(2.95") + 간격(0.1") = 3.05"
패널 bottom = 슬라이드 bottom(7.5") - 하단 여백(0.2") = 7.3"

→ 콘텐츠 영역(2.45"~7.5") 내 상하 여백 0.2" 균등 ✓
```

### 배지-패널 분리 원칙 (L09 동일)

```
배지(BEFORE/AFTER)는 메인 패널과 겹치지 않는다.

배지 top  = 본문설명글 bottom(2.45") + 상단 여백(0.2") = 2.65"
배지 bottom = 2.65" + 0.3" = 2.95"
패널 top  = 배지 bottom(2.95") + 간격(0.1") = 3.05"
패널 bottom = 슬라이드 bottom(7.5") - 하단 여백(0.2") = 7.3"

→ 콘텐츠 영역(2.45"~7.5") 내 상하 여백 0.2" 균등 ✓
```

### 한국 IT 업계 적용 근거

- **수치 중심 성과 보고**: 임원은 "좋아졌다"가 아닌 "3.2초→0.8초(75% 개선)"을 원함 — 24pt Bold로 즉시 인식 (design-spec.md § 3.7 BLUF)

- **Before/After 시간축**: NIA 5단 구성(현황→효과)의 효과(Effect) 단계에 직접 대응 (design-spec.md § 3.8)
- **비대칭 색상 톤**: 좌(BG_BOX 회색) → 우(E6F0FF 파랑)로 개선 방향을 색상으로 전달 — L09 동일 톤 체계
- **변화율 배지**: "▼ 75%" 한 줄로 5초 안에 핵심 성과 인식 가능 — 5-second test 통과 (design-spec.md § 3.2)

### 한국 IT 업계 적용 근거

- **RFP 제안서 신뢰성**: 장점만 나열하면 "과장"으로 인식 → 단점도 투명하게 제시하되 완화 방안까지 명시하여 리스크 관리 능력 입증 (design-spec.md § 3.6 차별성)
- **수치 기반 신뢰**: 모든 장단점에 정량 근거 필수 — "관리 부담 감소" X → "관리 부담 80% 감소 (AWS 실측)" O (design-spec.md § 3.5)
- **완화 방안 (Mitigation)**: 각 단점에 "→ 완화 방안" + PRIMARY 색상 → 단점이 치명적이지 않음을 시각적으로 입증
- **종합 판단 (Verdict)**: Pros/Cons만 나열하면 임원 보고 불완전 — **"도입 권장/보류/거부"** 결론 + Action Item까지 제시해야 3분 룰 충족 (design-spec.md § 3.7 BLUF)
- **항목 중요도**: Critical(★) 항목을 좌상단 배치하여 5초 안에 핵심 장점 인식 (design-spec.md § 3.2 시선 흐름)
- **대비 색상**: SUB_GREEN(긍정) vs SUB_ORANGE(부정) — 색맹 대응으로 아이콘(✅/⚠️)도 병행 (design-spec.md § 4.6)
- **NIA 5단 매핑**: 현황(Pros/Cons 분석) → 방향(Verdict 결론) 2단계를 1슬라이드에 완결 (design-spec.md § 3.8)

## 본문 레이아웃 상세: Do / Don't (`do_dont`) — L14

**목적**: 베스트 프랙티스(권장)와 안티패턴(비권장)을 대비하여 **실수 방지** + **올바른 대안(Alternative)** 제시.

L13(Pros & Cons)이 옵션 평가 + 결론이라면, L14는 **행동 지침** — "이렇게 하지 마라 → **대신 이렇게 하라**"
한국 IT 업계 적용: 아키텍처 가이드, 코딩 컨벤션, 운영 매뉴얼에서 "잘못된 사례"를 시각적으로 경고하되, **대안 없는 경고는 불완전** — 반드시 올바른 행동을 함께 제시 (design-spec.md § 3.5 문서 완결성).

참조: design-spec.md § 3.1 (Minto Pyramid), § 3.3 (시각 위계 3단계), § 3.4 (인지 부하 감소 — 시각화 우선), § 3.5 (고밀도 정보·문서 완결성), § 4.1 (WCAG AA 대비율), § 4.6 (색상 수 제한)
템플릿 참조 페이지: 없음 (신규 설계)

- **고밀도 정보**: 각 항목에 근거(detail) + 사례(incident) + 대안(alternative) — 빈 공간 없음 (design-spec.md § 3.5)

**목적**: 순환 프로세스 (4방향 루프) — 개발 → 빌드 → 테스트 → 배포 반복.

4개 박스를 원형으로 배치하고 화살표로 연결한 순환 구조.
한국 IT 업계 적용: CI/CD 파이프라인, 데이터 수집 사이클, PDCA(Plan-Do-Check-Act).

참조: design-spec.md § 3.4 (시각화 우선)
템플릿 참조 페이지: 없음 (신규 설계)

본문 제목 + 본문 설명글 있음 → **분기 A**

#### 구조

```
[중제목]  [설명글]
──────────────────────────────── (구분선)
[본문제목]                                          ← 16pt PRIMARY Bold
[본문설명]                                          ← 13pt DARK_GRAY

           ┌────────┐
           │  Step1 │  상단
           └────────┘
              ↓  ↑
    ┌────────┐    ┌────────┐
    │ Step4  │    │ Step2  │  좌우
    └────────┘    └────────┘
              ↓  ↑
           ┌────────┐
           │  Step3 │  하단
           └────────┘
```
